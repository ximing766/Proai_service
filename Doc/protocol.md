# ProAI 通信协议规范

本文档定义了 AI 模组内部的 **主服务 (Master Service)** 与 **从服务 (Slave Service)** 之间的通信机制，以及底层 **涂鸦串口协议** 的处理分工。

---

## 1. 架构与职责分工

为了保证系统稳定性和实时性，采用了主从分离架构。

| 特性 | 主服务 (Master Service) | 从服务 (Slave Service) |
| :--- | :--- | :--- |
| **语言** | Python / Go / C++ | C |
| **核心职责** | 业务逻辑、云端交互、AI 推理 | 硬件抽象、协议转换、保活 |
| **串口处理** | **不直接操作串口** | **独占管理 /dev/ttySx** |
| **协议感知** | 仅关注业务数据 (如: "开灯") | 处理物理帧 (头、校验、转义) |
| **稳定性** | 可能因业务逻辑复杂而重启 | 极高，作为守护进程常驻 |

**设计优势**：主服务只需要关注 IPC 协议，**完全不需要** 处理复杂的串口字节流、校验和计算或粘包问题。

---

## 2. 内部通信协议 (IPC)

主从服务之间通过 **TCP Socket (Port 5555)** 进行长连接通信。

### 2.1 协议帧格式 (JSON over TCP)

通信采用 Length-Prefixed JSON 格式（大端序），以提高可扩展性：

| 字段 | 长度 | 说明 |
| :--- | :--- | :--- |
| **Length** | 4 Bytes | 后续 JSON 字符串的字节长度 (Big Endian) |
| **JSON** | N Bytes | UTF-8 编码的 JSON 字符串 |

### 2.2 JSON 消息结构

所有消息均为 JSON 对象，包含 `type` 字段区分消息类型。

**通用结构**:
```json
{
    "type": "string",  // 消息类型
    "data": object     // 负载数据 (可选)
}
```

### 2.3 消息类型定义

| Type | 说明 | Data 结构 | 示例 |
| :--- | :--- | :--- | :--- |
| `heartbeat` | 心跳保活 | (无) | `{"type": "heartbeat"}` |
| `send_mcu` | Master->Slave 发送串口指令 | `cmd` (int), `payload` (string, optional) | `{"type": "send_mcu", "data": {"cmd": 8}}` (无 payload) |
| `evt_mcu` | Slave->Master 上报 MCU 数据 | `cmd` (int), `payload` (string, optional) | `{"type": "evt_mcu", "data": {"cmd": 7, "payload": "010001"}}` |
| `slave_status` | 状态报告 | `code` (int), `msg` (string) | `{"type": "slave_status", "data": {"code": 1, "msg": "UART Error"}}` |

## 3. 交互流程详解

### 3.1 场景一：系统初始化与心跳 (完全由从服务接管)
*   **MCU <-> Slave**: 从服务启动后，自动与 MCU 进行 `0x00` (心跳) 和 `0x01` (查询产品信息) 的交互。
*   **Master**: 不需要参与此过程。从服务只有在检测到 MCU 异常断开或重连时，才通过 `IPC_EVT_SLAVE_STATUS` 通知主服务。

### 3.2 场景二：主服务控制设备 (如：打开座椅加热)
1.  **Master**: 决定打开加热 (DPID=1, Bool=True)。
2.  **Master -> Slave (IPC)**: 发送 `send_mcu` 消息。
    *   JSON: `{"type": "send_mcu", "data": {"cmd": 6, "payload": "0101000101"}}`
    *   其中 `cmd=6` 代表命令下发，`payload` 为 DP 数据的十六进制字符串。
3.  **Slave**: 
    *   收到 IPC 消息，解析 JSON。
    *   **自动封装**: 添加帧头 `0x55AA`、版本号、计算校验和。
    *   **发送串口**: `55 AA 00 06 00 05 01 01 00 01 01 CS` -> MCU。

### 3.3 场景三：状态查询 (无 Payload 指令)
1.  **Master**: 需要主动查询当前所有 DP 状态。
2.  **Master -> Slave (IPC)**: 发送 `send_mcu` 消息，不带 payload。
    *   JSON: `{"type": "send_mcu", "data": {"cmd": 8}}`
    *   其中 `cmd=8` 为涂鸦协议的状态查询指令。
3.  **Slave**:
    *   解析发现无 payload，构造长度为 0 的串口帧。
    *   **发送串口**: `55 AA 00 08 00 00 CS` -> MCU。
4.  **MCU**: 收到查询后，通过 `0x07` 指令上报当前状态 (参考场景四)。

### 3.4 场景四：设备状态上报 (如：用户按下物理按键)
1.  **MCU -> Slave (UART)**: 发送状态上报帧 `55 AA 03 07 00 05 ... CS`。
2.  **Slave**:
    *   检测帧头，接收完整数据包。
    *   校验 CS 是否正确。如果不正确直接丢弃。
    *   **解包**: 提取核心数据 `0x07` + DP数据。
3.  **Slave -> Master (IPC)**: 发送 `evt_mcu` 消息。
    *   JSON: `{"type": "evt_mcu", "data": {"cmd": 7, "payload": "0701000101"}}`
4.  **Master**: 解析 JSON 中的 payload，更新业务状态或上报云端。

## 4. 底层串口协议参考 (Tuya)

> 以下内容仅供参考，由 **Slave Service** 内部实现，Master 开发人员无需关注细节。

### 7. 基础服务

#### 7.1. 心跳检测(0x00)
- 模组首次上电，与 MCU 握手需要发送心跳包。
- 模组与 MCU 在正常工作时，定期发送心跳包检测连接是否正常。
- 模组发送：
  - 帧头: 0x55aa
  - 版本: 0x00
  - 命令字: 0x00
  - 数据长度: 0x0000
  - 校验和
- MCU 返回：
  - 帧头: 0x55aa
  - 版本: 0x03
  - 命令字: 0x00
  - 数据长度: 0x0001
  - 数据: 0x00 (首次上电) / 0x01 (正常工作)

#### 7.2. 查询产品信息(0x01)
- 模组查询 MCU 的 PID (Product ID)、版本、模式等信息。
- MCU 返回 PID、版本号、模式配置。
- 模式配置字段 (m, mht, qr, slave, ext_ota, U, dp_cache, cops, dp_ack, online_tts, gps, env_pre, net_type)。

#### 7.3. 查询 MCU 设定模组的工作模式(0x02)
- 模组工作模式：
  - MCU 与模组配合处理：模组通过串口通知 MCU 状态，MCU 显示。
  - 模组自处理：模组通过 GPIO 驱动 LED。
- MCU 返回配置：配合处理(0x0000) 或 自处理(0x0002)。

#### 7.4. 报告设备联网状态(0x03)
- 模组向 MCU 报告联网状态：
  - 0x00: SIM 卡未连接
  - 0x01: 搜索网络中
  - 0x02: 已成功注册未联网
  - 0x03: 联网成功并获取到 IP
  - 0x04: 设备已连接到云端
  - 0x05: SIM 基站注册被拒绝
  - 0x06: 设备待配网
  - 0xFF: 未知状态

#### 7.5. 模组重置(0x04)
- MCU 通知模组重置，解除绑定。

#### 7.6. 获取版本信息(0x71 41)
- 获取固件版本信息。

### 8. DP 服务

#### 8.1. 数据单元
- DPID (1 byte)
- Type (1 byte): Raw(0), Bool(1), Value(2), String(3), Enum(4), Bitmap(5)
- Len (2 bytes)
- Value (N bytes)

#### 8.2. 命令下发 (0x06)
- 模组 -> MCU
- 包含一个或多个 DP 数据单元。

#### 8.3. 状态上报 (0x07)
- MCU -> 模组
- 异步上报 DP 状态变化。

#### 8.4. 状态查询 (0x08)
- 模组 -> MCU
- 查询所有 DP 状态。MCU 收到后通过 0x07 上报。

#### 8.5. 状态同步上报：非记录类型 (0x22)
- MCU -> 模组
- 同步指令，需等待模组响应 (0x23)。

#### 8.6. 状态同步上报：记录类型 (0x26)
- MCU -> 模组
- 带时间戳的数据上报。
- 需等待模组响应。

### 9. MCU 升级服务

#### 9.1. 升级启动(0x0a)
- 模组通知 MCU 开始升级，包含固件大小、MD5 等信息。
- MCU 回复同意或拒绝。

#### 9.2. 升级包传输(0x0b)
- 模组分包发送固件数据。
- MCU 回复 ACK。

#### 9.3. 升级状态通知(0x34 09)
- 模组查询或通知升级状态。
